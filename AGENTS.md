# AGENTS.md

LLM 코딩 실수를 줄이기 위한 행동 지침입니다. 필요하면 프로젝트별 지침과 함께 적용합니다.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선합니다. 사소한 작업에는 상황에 맞게 판단합니다.

## 1. 코딩 전에 생각하기

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러냅니다.**

구현 전에:

- 가정을 명시합니다. 확실하지 않으면 질문합니다.
- 여러 해석이 가능하면 그대로 제시하고 조용히 하나를 고르지 않습니다.
- 더 단순한 접근이 있으면 말합니다. 필요하면 반대 의견을 냅니다.
- 불명확한 것이 있으면 멈춥니다. 무엇이 혼란스러운지 말하고 질문합니다.

## 프로젝트 스킬

저장소 로컬 스킬은 `.agents/skills`에 둡니다.

- `frontend-architecture`는 React 구조, 레이어 배치, public API, import 경계 결정에 사용합니다.
- `korean-jsdoc-style`은 TypeScript/React JSDoc, TSDoc, 한국어 public API 문서, IDE hover DX 개선에 사용합니다.
- `web-graphics-editor-design-patterns`는 편집 가능한 캔버스, 그래픽 편집기, 어노테이션, 다이어그램, 화이트보드, undo/redo, 선택, 도구 모드 아키텍처에 사용합니다.

## 2. 단순함 우선

**문제를 해결하는 최소 코드만 작성합니다. 추측성 코드는 쓰지 않습니다.**

- 요청받지 않은 기능은 추가하지 않습니다.
- 한 번만 쓰는 코드에 추상화를 만들지 않습니다.
- 요청받지 않은 "유연성"이나 "설정 가능성"을 추가하지 않습니다.
- 불가능한 상황을 위한 오류 처리를 넣지 않습니다.
- 200줄로 쓴 코드가 50줄로 가능하다면 다시 씁니다.

스스로 묻습니다. "시니어 엔지니어가 이걸 과하다고 볼까?" 그렇다면 단순화합니다.

## 3. 외과적 변경

**필요한 부분만 건드립니다. 내가 만든 흔적만 정리합니다.**

기존 코드를 수정할 때:

- 인접한 코드, 주석, 포맷을 "개선"하지 않습니다.
- 고장 나지 않은 것을 리팩터링하지 않습니다.
- 내가 다르게 하고 싶더라도 기존 스타일을 따릅니다.
- 관련 없는 죽은 코드를 발견하면 언급만 하고 삭제하지 않습니다.

내 변경으로 고아 코드가 생겼을 때:

- 내 변경으로 사용되지 않게 된 import, 변수, 함수는 제거합니다.
- 기존에 이미 죽어 있던 코드는 요청받지 않았다면 제거하지 않습니다.

판단 기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 합니다.

## 4. 목표 중심 실행

**성공 기준을 정의하고, 검증될 때까지 반복합니다.**

작업을 검증 가능한 목표로 바꿉니다.

- "검증 추가" → "잘못된 입력 테스트를 작성하고 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고 통과시킨다"
- "X 리팩터링" → "전후 테스트가 통과하는지 확인한다"

여러 단계 작업에는 짧은 계획을 작성합니다.

```text
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

강한 성공 기준은 독립적으로 반복할 수 있게 합니다. 약한 기준("작동하게 만들기")은 계속 확인을 요구합니다.
